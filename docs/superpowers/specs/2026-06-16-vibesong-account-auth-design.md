# VibeSong Account Auth & Taste Persistence Design

Generated by brainstorming on 2026-06-16
Status: APPROVED FOR IMPLEMENTATION

## Problem Statement

VibeSong currently has no real accounts. Spotify login is optional and JWT-only (no database). Every other piece of state — the taste quiz answer, saved songs, skipped songs, and credits — lives in `localStorage`, which means it's per-browser, easy to reset, and unusable for any kind of "the app gets to know your taste over time" feature.

The user wants a mandatory login (Google or email) right after the taste quiz, and wants saved/skipped song behavior persisted so future matching can learn from it — not just the last dozen swipes, but an aggregate sense of what this person actually likes.

This pulls the Supabase-accounts part of Phase 2 forward into Phase 1, intentionally, at the user's request.

## Goals

- Require sign-in (Google or email magic link) immediately after the taste quiz step, before any photo upload.
- Keep the existing Spotify "Connect Spotify" enhancement flow working exactly as it does today.
- Persist taste quiz answers, save/skip history, and credits server-side (Supabase Postgres) instead of localStorage.
- One-time migration of existing localStorage data into the new account on first sign-in.
- Compute an aggregate taste profile (preferred/avoided genres and artists) from full save/skip history and feed it into the GPT matching prompt, supplementing — not replacing — the existing matching algorithm design.
- Move credits server-side so they can't be reset by clearing browser storage.

## Non-Goals

- No Stripe / paid credits in this pass (still Phase 1, no payments).
- No self-service account deletion in this pass (deferred; noted as a follow-up needed before real launch, since Spotify's developer terms require letting users delete their data on disconnect).
- No privacy policy page (legal/business document, not code).
- No changes to the Spotify "enhance" flow's behavior — it stays transient, per-request, exactly as implemented today.
- No per-track genre tagging from GPT — genre signal for taste learning comes from the photo's existing `musicDNA.genres`, snapshotted at save/skip time.

## Compliance Constraint (Spotify Developer Terms)

Per Spotify's developer rules: don't store Spotify content longer than necessary, don't use Spotify content to train AI/ML models, and let users disconnect/delete their data. To stay clearly inside these rules:

- **No raw Spotify API response data (top tracks, audio features, etc.) is ever written to Supabase.** The `/api/enhance` call remains transient and per-request, used only to re-rank that one batch of GPT candidates in memory.
- Everything persisted long-term (`user_taste`, `track_feedback`) is first-party VibeSong data: what the user typed into the taste quiz, and which AI-suggested songs they personally saved or skipped inside VibeSong.
- Google/Email sign-in doesn't touch Spotify's Development Mode user quota at all — only people who click "Connect Spotify" count against that limit, and that flow is unchanged.

## Architecture

### Auth

Extend the existing NextAuth v5 setup (`auth.ts`) with two more providers alongside the existing Spotify provider:

- **Google** — NextAuth's built-in Google provider. Needs a free OAuth Client ID/Secret from Google Cloud Console.
- **Email** — NextAuth's built-in magic-link provider. Needs an email-sending service (Resend recommended: free tier, simple API).

Magic-link email requires persisting one-time verification tokens, which means the session strategy moves from `"jwt"` to `"database"`, backed by the official `@auth/supabase-adapter`. This is a real change to `auth.ts`:

- Remove the JWT-only `jwt`/`session` callbacks that stash `account.access_token` into the token.
- With database sessions, the `session` callback receives `user` (not `token`). Spotify's access token is fetched on demand from the adapter's `accounts` table (`next_auth.accounts`, filtered by `userId` + `provider = 'spotify'`) when `/api/enhance` needs it, rather than being cached in a JWT.
- `session.user.id` becomes the database user id directly.

New env vars: `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `RESEND_API_KEY`, `EMAIL_FROM`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` (server-only).

### Database (Supabase Postgres)

**Auth tables** — standard `@auth/supabase-adapter` schema: `next_auth.users`, `next_auth.accounts`, `next_auth.sessions`, `next_auth.verification_tokens`. App code never queries these directly except via the NextAuth adapter.

**App tables** (`public` schema):

```sql
create table public.profiles (
  user_id uuid primary key references next_auth.users(id) on delete cascade,
  credits int not null default 3,
  migrated_local_data boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table public.user_taste (
  user_id uuid primary key references next_auth.users(id) on delete cascade,
  genres text[] not null default '{}',
  favorite_artists text[] not null default '{}',
  default_mood text not null default '',
  discovery_style text not null default 'balanced',
  dislikes text[] not null default '{}',
  language_preference text not null default 'No preference',
  energy_preference text not null default 'depends',
  setup_complete boolean not null default false,
  updated_at timestamptz not null default now()
);

create table public.track_feedback (
  id bigint generated always as identity primary key,
  user_id uuid not null references next_auth.users(id) on delete cascade,
  action text not null check (action in ('saved', 'skipped')),
  title text not null,
  artist text not null,
  reason text,
  match_score int,
  genres text[] not null default '{}',
  artwork text,
  thumbnail text,
  apple_music_url text,
  youtube_url text,
  youtube_id text,
  preview_url text,
  preview_provider text check (preview_provider is null or preview_provider in ('itunes', 'youtube')),
  source_image text,
  created_at timestamptz not null default now()
);

create index track_feedback_user_action_idx
  on public.track_feedback (user_id, action, created_at desc);
```

`genres` on `track_feedback` is the key field for taste learning: GPT doesn't tag individual tracks with a genre, so at save/skip time we snapshot the *photo's* `musicDNA.genres` the track was matched against. Tallying these across history approximates "this person saves dream-pop/indie-photo tracks and skips EDM-photo tracks" without an extra GPT call.

**Access model**: RLS enabled on all `public` tables with no policies (default deny for `anon`/`authenticated` roles — defense in depth). All actual access goes through server-side API routes using the Supabase **service role key**, which bypasses RLS by design and is never sent to the browser. The real security boundary is that every API route reads `session.user.id` from the authenticated NextAuth session and scopes its query to that id — not Postgres RLS.

## Product Flow

**First-time visitor:**
1. `/app`, no session, no local data → `TasteSetup` quiz (unchanged) → complete or "Skip for now."
2. New `AuthGate` screen appears: "Sign in with Google" or "Continue with email" (magic link). Upload UI is unreachable until signed in.
3. On sign-in: server creates `profiles` row (3 credits) and `user_taste` row from quiz answers via NextAuth's `signIn` event (no local data existed, nothing to migrate).
4. Lands on upload screen; credits/taste now sourced from Supabase via session.

**Returning visitor with existing localStorage data:**
1. Same quiz/`AuthGate` sequence.
2. On sign-in, server checks `profiles.migrated_local_data`. If `false`, `POST /api/migrate-local` reads whatever the client still has in `localStorage` (`userTaste`, `vibesong_library`, `vibesong_skipped`, `vibesong_credits`) and inserts it into `user_taste`, `track_feedback`, and `profiles.credits`, then sets `migrated_local_data = true`. Idempotent — running it again is a no-op.

**Already signed in, new visit:**
- Quiz and `AuthGate` both skipped; `user_taste.setup_complete` (from the server) replaces the `localStorage.getItem("userTaste")` check.

**Save/skip during swiping:**
- `useAppStore`'s `saveTrack`/`skipTrack` still update in-memory state for instant UI, and now also fire `POST /api/feedback` with `{ action, track, genres: vibeProfile.musicDNA.genres }`. Direct localStorage writes for these two are removed; Supabase is the source of truth. `loadSavedSongs` is replaced by a fetch from `GET /api/feedback`.

**Analyze request (`/api/analyze`):**
- Reads the session server-side (via NextAuth's `auth()` helper), fetches `user_taste` and recent `track_feedback` (capped at the most recent 300 rows) from Supabase — the client no longer sends raw feedback in the request body.
- New `buildAggregateTasteProfile()` in `lib/matching.ts` tallies genre/artist frequency from saves (positive) and skips (negative). A genre or artist is added to the "avoid" list only when it has been skipped at least 3 times AND the skip count is more than double the save count for that same genre/artist (`skipCount >= 3 && skipCount > saveCount * 2`). The top 5 most-saved genres/artists (by raw count, ties broken by most recent) become "learned preferences," merged with — not overriding — the explicit quiz answers from `user_taste`, and fed into the existing GPT prompt construction.
- If a user has no feedback history yet (new account), this returns the explicit quiz answers untouched — same as today's behavior.

**Credits:**
- `lib/credits.ts`'s localStorage functions are replaced by `GET /api/credits`, `POST /api/credits/deduct`, `POST /api/credits/add` (used by the existing mock pricing modal). All read/write `profiles.credits` for `session.user.id`.

## Error Handling

- `/api/analyze`, `/api/feedback`, `/api/credits/*` return `401` for any request without a valid NextAuth session.
- Magic-link send failures (email provider down, invalid address) surface as an inline error on the `AuthGate` form.
- `/api/migrate-local` is best-effort per row: a partial failure is logged server-side, and `migrated_local_data` is still set to `true` so the user isn't stuck retrying — worst case is losing some history, not the account.
- Empty/failed `track_feedback` fetch in `/api/analyze` falls back to explicit quiz answers only.

## Testing / Verification

- `npm run lint` and `npm run build` after each implementation task.
- Smoke test A (fresh user): new browser profile → quiz → Google sign-in → upload → save/skip several tracks → sign out → sign back in → confirm taste, credits, and library persisted.
- Smoke test B (migration): seed `localStorage` (`userTaste`, `vibesong_library`, `vibesong_skipped`, `vibesong_credits`) before first sign-in → confirm `/api/migrate-local` runs once, populates Supabase correctly, and a second sign-in does not duplicate rows.
- Confirm `/api/analyze` returns `401` when called without a session (e.g. via curl).
- Confirm an authenticated `/api/analyze` call with feedback history actually changes the merged taste profile sent to GPT (log it server-side during development to verify).
