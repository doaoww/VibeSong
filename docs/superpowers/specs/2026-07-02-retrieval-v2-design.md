# VibeSong Retrieval v2 — Photo Meaning → Song Catalog Bridge

**Date:** 2026-07-02
**Status:** Approved — ready for implementation planning

---

## Problem Statement

An architecture audit (2026-07-02) traced the live recommendation pipeline end to end and found the bottleneck is not the catalog. The catalog already carries a rich, closed-vocabulary tag layer (`story_intent_tags`, `modern_aesthetic_tags`, `mood_tags`, `story_context_tags`) generated by GPT-4o per song. The photo side is where signal is lost: GPT-4o's vision analysis already reasons about scene, social context, and narrative moment (`scene`, `people.socialVibe`, `momentType`, `vibeCaption`), but only a flat 10-dimension `emotional_vector` survives into retrieval. Everything else is returned to the client for display and discarded before matching.

The consequence: two photos with very different stories (a girl on a beach at sunset vs. a girl alone in a car at night) can produce near-identical emotional vectors, so pgvector cosine search treats them as musically interchangeable when they shouldn't be. Retrieval is also single-source (pgvector top-50 only), so a song that is a perfect story/context/taste match but not a close vector neighbor is never seen by scoring at all.

## Core Principle

A photo is not just a point in a 10-dimensional aesthetic space. It carries scene/context, narrative intent, aesthetic register, mood, and an implicit musical direction — all of which GPT-4o already infers today. Retrieval v2's job is to preserve those signals and give each one its own path into the candidate pool, instead of compressing everything through one vector before songs are even considered.

**Constraint carried over from earlier discussion:** no new OpenAI/embedding calls in this phase. Everything below extends the existing single GPT-4o vision call and existing catalog tag columns — no new song-side schema, no new AI calls, no new external API calls at request time.

## Signal Inventory — What a Photo Represents After This Change

| Signal | Source | New? |
|---|---|---|
| Emotional vector (10-dim) | GPT-4o vision (existing `photoVector`) | No |
| Scene/context tags | GPT-4o vision, closed vocab (expanded `STORY_CONTEXT_TAGS`) | New use of expanded existing field |
| Story intent tags | GPT-4o vision, closed vocab (`STORY_INTENT_TAGS`) | New — photo-side use of existing catalog taxonomy |
| Modern aesthetic tags | GPT-4o vision, closed vocab (`MODERN_AESTHETIC_TAGS`) | New |
| Mood tags | GPT-4o vision, closed vocab (`MOOD_TAGS`) | New |
| Anti-tags | GPT-4o vision, union of the 4 vocabs above | New |
| Music direction (genres/references/avoid) | GPT-4o vision, structured, open vocab | New |
| Energy bounds | GPT-4o vision, `{min, max}` | New, replaces fixed ±0.5 rule tolerance |
| Photo confidence | GPT-4o vision (existing `photoConfidence`) | No — newly used to gate all signals above |
| User taste | Onboarding + feedback history (existing) | No — newly gets its own retrieval pool, not just a query-vector blend input |

`social_read` (requested in discussion) is **not** a new field — it's the existing `momentType` enum, now used as reasoning context that shapes the other fields GPT emits (tighter `energy_bounds`, relevant `anti_tags`) rather than as an independently-matched field, since songs have no comparable column to match it against. Revisit as its own signal only if a future catalog pass adds a song-side equivalent.

## Layer 1: Photo Analysis — Extended Output Schema

Extends the existing single GPT-4o call in `app/api/analyze/route.ts`. No new call, no new `max_tokens` pressure of consequence (a few short arrays and two floats). Added alongside today's `photoVector`:

```json
"matchSignals": {
  "scene_context_tags":    ["night drive"],
  "story_intent_tags":     ["lonely but pretty"],
  "modern_aesthetic_tags": ["slavic underground"],
  "mood_tags":             ["melancholic", "dreamy"],
  "anti_tags":             ["euphoric", "party"],
  "music_direction": {
    "genres":     ["slavic indie", "moody r&b"],
    "references": ["The xx", "Lana Del Rey", "Molchat Doma"],
    "avoid":      ["EDM", "upbeat pop"]
  },
  "energy_bounds": { "min": 0.05, "max": 0.35 }
}
```

Field rules:
- `scene_context_tags`: 1-3 values, closed vocabulary = expanded `STORY_CONTEXT_TAGS` (see Layer 2). This is the "what kind of photo/moment is this, physically and socially" axis.
- `story_intent_tags`: 1-3 values, closed vocabulary = `STORY_INTENT_TAGS`. The narrative/emotional-intent axis (unchanged taxonomy, first time used photo-side).
- `modern_aesthetic_tags`: 0-3 values, closed vocabulary = `MODERN_AESTHETIC_TAGS`.
- `mood_tags`: 1-2 values, closed vocabulary = `MOOD_TAGS`.
- `anti_tags`: 0-4 values, drawn from the **union** of the three tag vocabularies above (not `scene_context_tags` — a photo being "not a beach photo" isn't a useful anti-signal; the point of anti-tags is ruling out incompatible moods/intents/aesthetics). Songs carrying any of these get hard-filtered — see Layer 4.
- `music_direction.genres`: 1-3 open-vocabulary genre/style strings (same free-text convention as `song.genre_tags`).
- `music_direction.references`: 0-3 real artist names GPT associates with this photo's vibe. **Hard invariant: references are retrieval/scoring signals only. No code path may use a reference to directly return or inject a specific song. Every candidate, regardless of how it entered the pool, passes through the full Rules + Scoring layer.**
- `music_direction.avoid`: 0-3 open-vocabulary genre/style strings to down-weight.
- `energy_bounds`: `{min, max}` floats in `[0,1]`. Server-side validated: if missing, malformed, non-finite, or `min > max`, fall back to `[energy-0.25, energy+0.25]` clamped to `[0,1]` (mirrors today's fixed-tolerance behavior).

All four closed-vocabulary fields are validated with the existing `splitByCanonical()` helper (`lib/tagTaxonomy.ts`) immediately after parsing GPT's response. Anything outside the canonical lists is dropped, never passed downstream — same defensive pattern already used for song auto-tagging.

**Naming convention note:** `matchSignals`'s fields are snake_case (`scene_context_tags`, not `sceneContextTags`) deliberately — this matches the existing `autoTagSong()` GPT prompt convention (`lib/autoTag.ts`), which already returns catalog tag fields as snake_case JSON that flows into TS objects unchanged. The rest of the photo-analysis prompt (`vibeCaption`, `photoVector`, etc.) is camelCase; `matchSignals` breaks from that specifically because its fields are the same tag vocabulary as the catalog side, and keeping the casing identical avoids a silent rename between "the tag GPT assigned to the photo" and "the tag GPT assigned to the song." The `/api/recommend` request body (Schema/API section below) stays camelCase, consistent with that route's existing fields (`photoVectorArray`, `vibeBoosts`, etc.).

### Confidence Gate

`photoConfidence` (existing field) gates every signal introduced in this spec, not just the emotional vector:
- `contextFit` / `vibeAestheticFit` / photo-sourced `storyFit` scoring contributions (Layer 5) are multiplied by `0.5 + clamp(confidence, 0, 1) × 0.5` — never fully zeroed, but capped down when GPT itself is unsure.
- `anti_tags` only apply as a hard filter when `photoConfidence >= 0.4`. Below that threshold they are dropped from the request entirely (Rules Layer filters are binary — a shaky read should not be able to silently remove otherwise-good candidates).
- `energy_bounds` widens toward the safe default (`[energy-0.25, energy+0.25]`) proportionally as confidence drops below 0.6.
- `music_direction.genres` / `.avoid` genre-score contributions (Layer 5) are scaled by the same `0.5 + confidence × 0.5` factor.

## Layer 2: Catalog Taxonomy Change

One small, deliberate taxonomy expansion in `lib/tagTaxonomy.ts`:

```ts
export const STORY_CONTEXT_TAGS = [
  "mirror selfie", "sunset", "night drive", "cafe", "car selfie",
  "gym", "beach", "city walk", "party", "outfit check",
  "travel", "group photo",          // NEW
] as const;
```

"Night city" (mentioned in discussion) is intentionally **not** added as a separate value — GPT is instructed to map it to the closest existing tag (`night drive` or `city walk`) to avoid near-duplicate categories that would fragment an already sparse tag (see catalog stats below). No other vocabulary changes. No new columns on `songs` — this spec reuses `story_context_tags`, `story_intent_tags`, `modern_aesthetic_tags`, `mood_tags`, `genre_tags` exactly as they exist today.

**Current catalog coverage** (checked live, 600 songs): 99.5% have `story_intent_tags`/`modern_aesthetic_tags`; only 45.5% (273/600) have `story_context_tags`/`vibe_summary` populated — those columns were added after most of the catalog was tagged. Existing distribution is heavily skewed: `night drive` (226), `city walk` (229), `party` (117) dominate; `cafe` (2), `outfit check` (3), `gym` (7) are near-empty. This matters for Layer 3's Context/Scene Pool — see Backfill section.

## Layer 3: Hybrid Retrieval — Four Candidate Pools

Four independent retrieval sources, merged and deduped by `id` before Rules + Scoring run — unchanged in shape from today's pipeline, just fed a richer candidate set.

```
                    ┌─────────────────────┐
                    │   matchSignals +     │
                    │   query vector +     │
                    │   user taste profile │
                    └──────────┬───────────┘
        ┌───────────┬──────────┼──────────┬────────────┐
        ▼           ▼          ▼          ▼
  [Photo Vector] [Story Tags] [Context/Scene] [Taste]
   pgvector       tag overlap  tag overlap    artist/genre
   top 25         top 25       top 20         match, top 20
        └───────────┴──────────┴──────────────┘
                         ▼
                  merge + dedupe by id
                  (~60-80 unique candidates)
                         ▼
                  Rules Layer (unchanged shape,
                  new energy_bounds + anti_tags source)
                         ▼
                  Scoring Layer (existing formula +
                  2 new components, confidence-gated)
                         ▼
                    Top 8-12 songs
```

**Pool 1 — Photo Vector Pool** (existing, trimmed): `match_songs` RPC, pgvector cosine similarity over `emotional_vector`, top 25 (down from 50, since three more pools now contribute).

**Pool 2 — Story Tags Pool** (new): overlap on `story_intent_tags`, `modern_aesthetic_tags`, `mood_tags` against the photo's respective `matchSignals`, top 25.

**Pool 3 — Context/Scene Pool** (new): overlap on `story_context_tags` against the photo's `scene_context_tags`, top 20.

Pools 2 and 3 share one SQL function, called twice with different arguments populated (only intent/aesthetic/mood filled for Pool 2, only context filled for Pool 3) rather than two near-identical functions:

Mirrors `match_songs`'s existing `RETURNS TABLE(...)` pattern exactly (not `RETURNS SETOF songs` — `lib/db/songs.ts` notes PostgREST can't resolve the `vector` column type through schema-cache introspection on the raw table, which is why every RPC declares an explicit typed column list instead). Parameters are `p_`-prefixed — discovered during implementation that unprefixed `aesthetic_tags`/`mood_tags` parameters collide with this function's own `RETURNS TABLE` output columns of the same name, which PL/pgSQL rejects with error 42P13 ("parameter name ... used more than once"). `match_songs`'s own `query_vector`/`match_count` avoid this only because neither name matches one of its output columns:

```sql
CREATE OR REPLACE FUNCTION public.match_songs_by_tags(
  p_context_tags   text[] DEFAULT '{}',
  p_intent_tags    text[] DEFAULT '{}',
  p_aesthetic_tags text[] DEFAULT '{}',
  p_mood_tags      text[] DEFAULT '{}',
  p_match_count    int DEFAULT 25
)
RETURNS TABLE (
  id uuid, title text, artist text, language text, energy float,
  popularity_tier int, emotional_vector vector(10), genre_tags text[],
  aesthetic_tags text[], mood_tags text[], story_intent_tags text[],
  modern_aesthetic_tags text[], story_context_tags text[],
  final_confidence float, needs_review boolean, itunes_preview_url text,
  artwork_url text, apple_music_url text, youtube_id text,
  quality_score float, distance float
)
LANGUAGE plpgsql AS $$
BEGIN
  RETURN QUERY
  SELECT
    s.id, s.title, s.artist, s.language, s.energy, s.popularity_tier,
    s.emotional_vector, s.genre_tags, s.aesthetic_tags, s.mood_tags,
    s.story_intent_tags, s.modern_aesthetic_tags, s.story_context_tags,
    s.final_confidence, s.needs_review, s.itunes_preview_url, s.artwork_url,
    s.apple_music_url, s.youtube_id, s.quality_score, NULL::float AS distance
  FROM public.songs s
  WHERE s.emotional_vector IS NOT NULL
    AND (
      (cardinality(p_context_tags)   > 0 AND s.story_context_tags    && p_context_tags)
      OR (cardinality(p_intent_tags)    > 0 AND s.story_intent_tags    && p_intent_tags)
      OR (cardinality(p_aesthetic_tags) > 0 AND s.modern_aesthetic_tags && p_aesthetic_tags)
      OR (cardinality(p_mood_tags)      > 0 AND s.mood_tags            && p_mood_tags)
    )
  ORDER BY s.quality_score DESC, s.id
  LIMIT p_match_count;
END;
$$;
```

Ranking inside this RPC is intentionally cheap (`quality_score desc`) — real ranking happens in the existing app-side Scoring Layer regardless of which pool a candidate arrived from, so the SQL only needs to surface plausible candidates, not order them precisely.

**Pool 4 — Taste Pool** (new): songs by a liked artist, a `music_direction.references` artist, or with positive genre overlap:

Prefixed `p_` for consistency with `match_songs_by_tags` above (no actual name collision here, but keeping both new functions' parameter conventions identical avoids relying on which specific columns happen to collide today):

```sql
CREATE OR REPLACE FUNCTION public.match_songs_by_taste(
  p_artist_patterns  text[] DEFAULT '{}',  -- pre-wrapped with %...% wildcards by the app layer
  p_positive_genres  text[] DEFAULT '{}',
  p_match_count      int DEFAULT 20
)
RETURNS TABLE (
  id uuid, title text, artist text, language text, energy float,
  popularity_tier int, emotional_vector vector(10), genre_tags text[],
  aesthetic_tags text[], mood_tags text[], story_intent_tags text[],
  modern_aesthetic_tags text[], story_context_tags text[],
  final_confidence float, needs_review boolean, itunes_preview_url text,
  artwork_url text, apple_music_url text, youtube_id text,
  quality_score float, distance float
)
LANGUAGE plpgsql AS $$
BEGIN
  RETURN QUERY
  SELECT
    s.id, s.title, s.artist, s.language, s.energy, s.popularity_tier,
    s.emotional_vector, s.genre_tags, s.aesthetic_tags, s.mood_tags,
    s.story_intent_tags, s.modern_aesthetic_tags, s.story_context_tags,
    s.final_confidence, s.needs_review, s.itunes_preview_url, s.artwork_url,
    s.apple_music_url, s.youtube_id, s.quality_score, NULL::float AS distance
  FROM public.songs s
  WHERE s.emotional_vector IS NOT NULL
    AND (
      (cardinality(p_artist_patterns) > 0 AND s.artist ILIKE ANY (p_artist_patterns))
      OR (cardinality(p_positive_genres) > 0 AND s.genre_tags && p_positive_genres)
    )
  ORDER BY s.quality_score DESC, s.id
  LIMIT p_match_count;
END;
$$;
```

`p_artist_patterns` = `taste.favoriteArtists ∪ photo.music_direction.references`, each wrapped `%name%` before the call. For v1 this is **substring/exact matching only** — no live Last.fm similar-artist expansion at request time, since that would add a real external API round-trip per recommendation (in tension with "no new cost/latency yet"). A `references` artist not literally present in the catalog contributes nothing this phase. Last.fm-based expansion (or a precomputed artist-similarity edge stored at tagging time) is a natural Phase 2 addition, not required for this spec to deliver value.

**Cold start:** a user with no liked artists, no `favorite_story_songs`, and no positive `genre_scores` simply gets an empty Taste Pool — no special-case code needed, the SQL naturally returns zero rows for empty input arrays.

**"Reranking":** the merged four-pool candidate set flows into the existing Rules Layer and Scoring Layer unchanged in structure (Layer 4/5 below extend the formula, not the architecture). No new reranking mechanism or model is introduced — the enriched Scoring Layer *is* the rerank step.

### Pool Debug Log

To measure whether hybrid retrieval is actually helping (as opposed to the tag/taste pools mostly re-surfacing what the vector pool already found), `/api/recommend` logs one aggregate object per request, alongside the existing per-song `debugLog`:

```json
{
  "vectorPoolCount": 25,
  "storyPoolCount": 18,
  "contextPoolCount": 12,
  "tastePoolCount": 9,
  "mergedCandidateCount": 47,
  "removedByRulesCount": 6
}
```

`vectorPoolCount`/`storyPoolCount`/`contextPoolCount`/`tastePoolCount` are each pool's raw row count *before* dedup (so overlap between pools is visible: `sum(pool counts) - mergedCandidateCount` = how much the pools actually agreed with each other — a low overlap number is the signal that hybrid retrieval is surfacing candidates the vector pool alone would have missed). `mergedCandidateCount` is the unique count after dedup, feeding the Rules Layer. `removedByRulesCount` is how many of those were dropped by Rules Layer filters (Layer 4) before scoring. This is computed in `app/api/recommend/route.ts` around the existing pool-merge and `buildRecommendations` calls — no new logging infrastructure, same `console.log` pattern already used for the per-song debug log.

## Layer 4: Rules Layer Changes

Applied in the existing order, after pool merge/dedupe, before scoring. Two changes to `lib/recommend.ts`:

1. **Energy compatibility** — replace the fixed `Math.abs(song.energy - queryEnergy) > 0.5` tolerance with a photo-aware tolerance, floored so an overly narrow GPT-provided `energy_bounds` can't over-filter:
   ```
   energyTolerance = max(0.2, (energyBounds.max - energyBounds.min) / 2)
   remove if Math.abs(song.energy - queryEnergy) > energyTolerance
   ```
   A photo GPT reads as having a narrow acceptable energy range (a still, quiet moment) filters more aggressively than one it reads as energy-ambiguous, but never tighter than a `0.2` half-width — protects against a single bad GPT read (e.g. `{min: 0.2, max: 0.22}`) wiping out most of the candidate pool on the energy rule alone.
2. **Anti-tags** — existing anti-tag filter (`req.antiTags`) now also receives `matchSignals.anti_tags` as a source, merged alongside the existing requested-vibe/onboarding-avoid sources, gated by the confidence threshold in Layer 1.

Edge case: if a tag ends up in both `req.storyIntentTags` (positive scoring signal) and `req.antiTags` (hard filter) — shouldn't happen if GPT is internally consistent, but it's an LLM — the anti-tag hard filter takes precedence; the song is removed regardless of any positive tag overlap.

## Layer 5: Scoring Layer Changes

Two new named components added to the existing formula in `lib/recommend.ts`, following the same style as today's logged components:

```
photoFit          = cosine(query_vector, song.emotional_vector) × 40
tasteFit          = genreOverlapScore × 15 + artistProximityScore × 10 + aestheticPresence × 5    (max 30)
storyFit          = min(3, storyIntentTagMatches) × 7                                              (max 21)
contextFit        = min(2, contextTagMatches) × 6 × confidenceFactor           [NEW]                (max 12)
vibeAestheticFit  = min(2, aestheticOrMoodMatches) × 5 × confidenceFactor      [NEW]                (max 10)
noveltyFit        = discoveryScore × 10                                                            (max 10)
qualityBonus      = song.quality_score × 5                                                         (max 5)

confidenceFactor  = 0.5 + clamp(photoConfidence, 0, 1) × 0.5

raw_score  = photoFit + tasteFit + storyFit + contextFit + vibeAestheticFit + noveltyFit + qualityBonus
final_score = clamp(raw_score - penalties, 0, 100)     (penalties unchanged from today)
```

Where `contextTagMatches` = count of overlapping values between `photo.matchSignals.scene_context_tags` and `song.story_context_tags`, and `aestheticOrMoodMatches` = count of overlapping values between (`photo.matchSignals.modern_aesthetic_tags` ∪ `photo.matchSignals.mood_tags`) and (`song.modern_aesthetic_tags` ∪ `song.mood_tags`) — a single combined overlap count across both tag pairs, not two separate terms, to keep the formula from growing an unbounded number of components as more tag types are added later.

Raw max rises from ~106 to ~128 before clamping; `photoFit`'s share of the theoretical max drops from ~38% to ~31%. This is the intended effect of the whole spec — diluting reliance on the flat 10-dim vector in favor of the richer signals now available.

Three signal sources feed *existing*, unmodified scoring functions rather than new ones:
- `storyFit` — now sourced from `photo.matchSignals.story_intent_tags` in addition to (future) requested-vibe tags, via the existing `req.storyIntentTags` field. No formula change.
- `tasteFit`'s `artistProximityScore` — `req.likedArtists` becomes `taste.favoriteArtists ∪ photo.music_direction.references` for this request only (not persisted to the user's stored taste profile). No formula change.
- `tasteFit`'s `genreOverlapScore` — `req.genreScores` becomes the user's stored `genre_scores` merged with `+weight` entries for each `music_direction.genres` term and `-weight` entries for each `music_direction.avoid` term (same signed-sum convention the onboarding avoid-list already uses). No formula change, confidence-gated per Layer 1.

`contextFit` and `vibeAestheticFit` are genuinely new score components (no existing analog — today's `aestheticMatch` inside `tasteFit` is a bare presence check, `song.aesthetic_tags.length > 0 ? 0.5 : 0`, not an overlap comparison; it is left as-is since it operates on a different, unrelated tag field, `aesthetic_tags`, not `modern_aesthetic_tags`).

Debug log (`DebugEntry`/`ScoreComponents` in `lib/recommend.ts`) gains `contextFit` and `vibeAestheticFit` fields, logged per song exactly like every other component today — this is not optional, per the existing "without this log it's impossible to debug why a recommendation was bad" principle.

## Query Vector Construction — Confidence-Aware Blend Fix

Independent of the pool/tag work above, but part of this spec since it's the same root issue (photo signal not weighted correctly): `blendQueryVector` (`lib/vectorMath.ts`) currently uses a fixed `0.55/0.45` photo/taste split regardless of how confident GPT was. Meanwhile `blendVectors` (used only to persist the long-term taste profile) already scales the split by confidence: `photoWeight = 0.2 + confidence × 0.5`.

**Fix:** `blendQueryVector`'s 2-signal path (no requested vibe — the only live path today) adopts the same confidence-driven formula:

```
photoWeight = 0.2 + clamp(photoConfidence, 0, 1) × 0.5   // range [0.2, 0.7]
tasteWeight = 1 - photoWeight
query_vector[i] = photo[i] × photoWeight + taste[i] × tasteWeight
```

This requires forwarding `photoConfidence` from `/api/analyze`'s response (already computed, not currently sent onward) through `app/page.tsx` into the `/api/recommend` request body. The 3-signal path (with requested vibe) is out of scope — that feature is still unwired to any UI per the earlier audit — and keeps its documented `0.40/0.25/0.35` formula for whenever it ships.

## Schema / API Changes Required

- **No new columns on `songs`.** `story_context_tags` vocabulary gains 2 values (Layer 2).
- **New GIN indexes** (currently missing, needed for `&&` overlap queries to be fast at 600+ rows): `story_context_tags`, `story_intent_tags`, `modern_aesthetic_tags`, `mood_tags`, `genre_tags`. Plus a btree index on `artist` if one doesn't already exist, for the Taste Pool's `ilike any` lookup.
- **New RPCs:** `match_songs_by_tags`, `match_songs_by_taste`.
- **New `lib/db/songs.ts` functions:** `searchCatalogByTags(...)`, `searchCatalogByTaste(...)`, mirroring the existing `searchCatalog`.
- **`RecommendRequest` interface** (`lib/recommend.ts`): add `energyBounds: {min: number, max: number}`, `photoConfidence: number`.
- **`/api/recommend` request body:** add `photoConfidence`, `sceneContextTags`, `storyIntentTags` (already exists, now also populated from the photo), `aestheticTags`, `moodTags`, `antiTags` (already exists, now also populated from the photo), `musicDirection: {genres, references, avoid}`, `energyBounds`. All optional/defaulted so a stale client doesn't break the route during rollout.
- **`app/page.tsx`:** forward the new `matchSignals` block and `photoConfidence` from the `/api/analyze` response into the `/api/recommend` call.
- **`/api/recommend` response:** add `poolStats` (see Pool Debug Log, Layer 3) alongside the existing `songs`, `totalCandidates`, `debugLog` fields.

## Catalog Backfill (Parallel, Non-Blocking)

327 songs (55%) lack `story_context_tags`/`vibe_summary` because those columns were added after they were tagged. Fix: re-run the existing `autoTagSong()` pipeline on songs where `story_context_tags` is empty — same pipeline already used for new songs, pointed at the backlog, now also aware of the 2 expanded taxonomy values. This does not block shipping Phase 1: an untagged song simply can't be surfaced by Pool 2/3 yet, but remains reachable via Pool 1 (vector) and Pool 4 (taste) as it is today.

## Explicit Invariants

1. `music_direction.references` (and any Taste Pool artist match) never bypasses Rules + Scoring to become a recommendation directly — every candidate from every pool is scored identically.
2. All four closed-vocabulary `matchSignals` fields are validated against their canonical lists server-side; out-of-vocabulary values are dropped, never scored or filtered on.
3. `anti_tags` only take effect as a hard filter above the `photoConfidence >= 0.4` threshold.
4. If `storyIntentTags` and `antiTags` conflict on the same tag, `antiTags` wins (song removed).

---

## What Does NOT Change

- Auth, credits, YouTube/iTunes playback — untouched.
- `lib/recommend.ts`'s existing rule order, penalty formulas, and `photoFit`/`noveltyFit`/`qualityBonus` calculations — untouched.
- Requested-vibe UI wiring — still unbuilt, still out of scope.
- Spotify integration — still dead code, untouched.
- `vibeCaption`/`vibe_summary` embedding-based semantic search — deferred; this spec's constraint is no new AI/embedding calls.
- The legacy, unused scoring system in `lib/matching.ts` — still dead code from the pre-redesign architecture, left in place; worth a separate cleanup pass, not part of this spec.

## Phasing

**Phase 1 (this spec, single implementation unit):**
1. Extend `app/api/analyze/route.ts` prompt + schema with `matchSignals` (Layer 1).
2. Expand `STORY_CONTEXT_TAGS` (Layer 2).
3. Add `match_songs_by_tags`, `match_songs_by_taste` RPCs + indexes (Layer 3, Schema).
4. Wire 4-pool merge/dedupe into `/api/recommend` (Layer 3).
5. Rules Layer: `energyBounds`, photo-sourced `anti_tags` (Layer 4).
6. Scoring Layer: `contextFit`, `vibeAestheticFit`, confidence gating, merged `genreScores`/`likedArtists` (Layer 5).
7. Confidence-aware `blendQueryVector` fix (Query Vector section).
8. Forward new fields through `app/page.tsx`.
