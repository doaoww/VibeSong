# VibeSong Matching Algorithm Design

Generated by brainstorming on 2026-06-16
Status: APPROVED FOR IMPLEMENTATION

## Problem Statement

VibeSong's current matching flow can choose emotionally plausible songs, but it relies too heavily on one GPT prompt and then resolves playback through YouTube. This creates two quality problems: recommendations can feel generic, and YouTube can return clips, intros, edits, or music videos that weaken the first-listen vibe.

The new matcher must make the user feel like the app understands both the photo and their personal music taste. The taste setup remains optional, but when users answer it, the algorithm should use it deeply.

## Goals

- Match both the uploaded photo and the user's taste in a balanced way.
- Analyze photos more deeply, including people, visible emotions, scene, activity, lighting, aesthetic, and social mood.
- Let users choose what "unique" means to them through a discovery-style quiz answer.
- Prefer tasteful, niche, or hidden-gem tracks without treating famous artists as bad.
- Add clean iTunes audio previews before falling back to YouTube.
- Improve YouTube fallback so clips, shorts, trailers, edits, reactions, and low-quality result types are rejected.
- Store save/skip feedback so future matches can become more personalized.

## Non-Goals

- No Last.fm integration in this pass.
- No Spotify dependency in this pass.
- No real-time trend engine in this pass.
- No database-backed machine learning; feedback can remain localStorage-based for now.

## Product Behavior

The taste quiz is optional. It adds richer fields:

- favorite genres
- favorite artists
- usual mood
- discovery style
- dislikes
- language or region preference
- energy preference

Discovery style supports four modes:

- `hidden-gems`: known or respected artists, less obvious songs
- `niche`: smaller artists and more underground discoveries
- `balanced`: a mix of familiar and surprising songs
- `popular-ok`: popular songs are allowed when the photo fit is excellent

The default is `balanced`.

## Matching Flow

1. User uploads a photo or video frame.
2. `/api/analyze` receives the image and optional `userTaste`.
3. GPT-4o performs deep visual analysis and generates a larger candidate pool, not just final cards.
4. Each candidate includes separate scoring fields:
   - `photoFitScore`
   - `tasteFitScore`
   - `discoveryFitScore`
   - `obviousnessPenalty`
   - `finalScore`
   - `reason`
   - `viralMomentSeconds`
5. Server normalizes and sorts candidates.
6. `/api/search-tracks` resolves the top candidates to clean playback:
   - iTunes Search API first for `previewUrl`
   - YouTube fallback when no iTunes preview exists
7. Swipe card plays the iTunes preview if available, otherwise YouTube.
8. Saves and skips update local feedback for future prompts.

## Scoring Philosophy

The algorithm should be balanced:

- photo fit matters strongly
- taste fit matters strongly
- discovery style changes how much obvious songs are penalized
- famous artists are allowed
- lazy song choices are penalized

Default score formula:

```txt
finalScore =
  photoFitScore * 0.40
+ tasteFitScore * 0.35
+ discoveryFitScore * 0.15
+ previewQualityScore * 0.10
- obviousnessPenalty
```

The implementation can use GPT's `finalScore` initially, then apply server-side adjustment after preview resolution.

## Data Model

`UserTaste` expands to:

```ts
interface UserTaste {
  genres: string[];
  favoriteArtists: string[];
  defaultMood: string;
  discoveryStyle: "hidden-gems" | "niche" | "balanced" | "popular-ok";
  dislikes: string[];
  languagePreference: string;
  energyPreference: "calm" | "medium" | "high" | "depends";
  setupComplete: boolean;
}
```

`Track` expands to include optional preview metadata:

```ts
interface Track {
  title: string;
  artist: string;
  reason: string;
  matchScore: number;
  viralMomentSeconds?: number;
  previewUrl?: string;
  previewProvider?: "itunes" | "youtube";
  artwork?: string;
  appleMusicUrl?: string;
  youtubeId?: string;
  youtubeUrl?: string;
  thumbnail: string;
}
```

## Error Handling

- If iTunes fails, continue to YouTube.
- If YouTube fails, skip that track.
- If fewer than five playable tracks are resolved, return partial results with status `206`.
- If the expanded taste object is missing older fields, defaults are applied server-side.
