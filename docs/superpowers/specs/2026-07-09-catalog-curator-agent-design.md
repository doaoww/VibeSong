# Catalog Curator Agent Design

Generated by brainstorming on 2026-07-09
Status: APPROVED FOR IMPLEMENTATION

## Problem Statement

The catalog only grows when someone manually writes a `scripts/seed-*.mjs` file with hand-picked `{ title, artist }` pairs and runs it against `/api/admin/songs`. That works, but it's a human bottleneck: the catalog goes stale unless someone remembers to seed new trending songs, and every seed script is a one-off manual chore.

The `/api/admin/songs` route already does all the hard work per song — dedupe against the catalog, then `autoTagSong` (iTunes + Last.fm + GPT-4o) to produce a fully-tagged, embedding-ready row via `insertSong`. What's missing is a source of *new candidates* and a scheduler to feed them in automatically. This design adds an autonomous agent that discovers trending tracks and keeps the catalog fresh without a human writing or running a seed script.

## Goals

- Run unattended, on a daily schedule, with no human approval step per song (full autonomy — the existing dedupe-by-title+artist behavior in `insertSong`/`findSongByTitleArtist` is the safety net, same as it is for manual seeding today).
- Pull candidates from a free, keyless, official source: Apple's public "most played" RSS/JSON feed, per country.
- Cover the same language/country spread already used in the hand-written seed scripts (English, Russian, French, Spanish, at minimum).
- Reuse the existing tagging pipeline (`autoTagSong` + `insertSong`) exactly as-is — no parallel/duplicate tagging logic.
- Bound worst-case daily OpenAI spend with a hard cap on new insertions per run, regardless of how many chart entries turn out to be novel.
- Survive partial failures: one bad chart entry (not found on iTunes/Last.fm, GPT hiccup, etc.) must not abort the rest of the run.
- Produce a run summary (`inserted` / `skipped` / `failed`) visible in Vercel's cron logs, since there is no review queue to check instead.

## Non-Goals

- No approval queue / staging table for candidates before insertion — this design is deliberately "fully autonomous," not "propose and wait."
- No TikTok/Instagram trending-audio API integration — those either require paid third-party providers or scraping; Apple's chart feed is judged close enough as a trending-music proxy and is free and stable.
- No retirement/pruning of stale catalog entries — out of scope, catalog only grows.
- No per-country/per-run configurability via UI — the country list and daily cap are code constants, changed by editing code, not through an admin screen.
- No changes to `autoTagSong`, `insertSong`, or the tagging pipeline itself — this design only adds a new candidate source and scheduler in front of the existing, unmodified pipeline.

## Architecture

**Scheduler**: `vercel.json` (new file) declares a single cron entry hitting `/api/cron/curate-catalog` once daily. Vercel Cron calls the route with an `Authorization: Bearer $CRON_SECRET` header automatically when `CRON_SECRET` is set as a project env var; the route checks that header the same way `isAdmin` in `app/api/admin/songs/route.ts` checks `x-admin-secret`.

**New route**: `app/api/cron/curate-catalog/route.ts`
- `export const runtime = "nodejs"` (consistent with the rest of the AI/data routes).
- Verifies the cron secret; `401` if missing/wrong.
- Calls `curateCatalog()` from the new lib module and returns its summary as JSON.
- No `GET` vs `POST` distinction needed beyond what Vercel Cron sends (`GET`).

**New lib module**: `lib/curator.ts`
- `TRENDING_COUNTRIES = ["us", "ru", "fr", "es", "gb"]` (matches the language spread in existing seed scripts: English, Russian, French, Spanish).
- `MAX_NEW_SONGS_PER_RUN = 15` — hard cap on inserts for the whole run, spent first-come across the country list in order.
- `AUTOTAG_MIN_INTERVAL_MS = 2000` — every `scripts/seed-*.mjs` runner throttles calls into the tagging pipeline to at least 2s apart to respect iTunes/Last.fm/GPT-4o rate limits (see `scripts/seed-catalog.mjs`'s `sleep`/`wait` logic). `curateCatalog` applies the same floor between candidates that actually reach `autoTagSong` (dedupe-skipped candidates are exempt, since they never call it) — otherwise an autonomous daily run risks the exact rate-limit failures the manual scripts were written to avoid.
- `fetchTrendingTracks(countryCode: string): Promise<{ title: string; artist: string }[]>` — fetches `https://rss.marketingtools.apple.com/api/v2/{countryCode}/music/most-played/50/songs.json` (verified live; the older `rss.applemarketingtools.com` host now 301-redirects here, so the canonical host is used directly), maps the feed's `feed.results[].{name,artistName}` into `{ title, artist }`, returns the top ~25.
- `curateCatalog(): Promise<{ inserted: {title,artist,id}[]; skipped: number; failed: {title,artist,error}[] }>`:
  1. For each country in `TRENDING_COUNTRIES`, in order: `fetchTrendingTracks`.
  2. For each candidate, in order: if `inserted.length >= MAX_NEW_SONGS_PER_RUN`, stop entirely.
  3. `findSongByTitleArtist(title, artist)` — if found, increment `skipped`, continue (no GPT call spent).
  4. Otherwise `autoTagSong(title, artist)` then `insertSong(tagged)`, wrapped in try/catch — success appends to `inserted`; failure appends `{title, artist, error: message}` to `failed` and continues to the next candidate (one bad track never aborts the run).
  5. Return the summary.

This directly reuses `autoTagSong` from `lib/autoTag.ts` and `findSongByTitleArtist` / `insertSong` from `lib/db/songs.ts` — the same three functions `app/api/admin/songs/route.ts` already calls. The cron route calls `curateCatalog()` in-process; it does not make an HTTP call to `/api/admin/songs`, since it's already running server-side with the same access to those functions.

## Data Flow

```
Vercel Cron (daily)
  -> GET /api/cron/curate-catalog  (Authorization: Bearer CRON_SECRET)
    -> curateCatalog()
      for each country in [us, ru, fr, es, gb]:
        -> fetchTrendingTracks(country)  [Apple RSS, no key]
        for each {title, artist} candidate (until cap reached):
          -> findSongByTitleArtist   [cheap dedupe, no GPT cost]
             found  -> skip
             not found ->
               -> autoTagSong(title, artist)  [iTunes + Last.fm + GPT-4o, unchanged]
               -> insertSong(tagged)
               success -> record inserted
               failure -> record failed, continue
      -> return { inserted, skipped, failed }
```

## Error Handling

- **Per-candidate isolation**: the `autoTagSong`/`insertSong` call for each candidate is individually try/caught inside the loop. A single failure (track not resolvable on iTunes/Last.fm, transient GPT error, etc.) is recorded in `failed` and the loop continues — matches the existing tolerance already implicit in manual seeding (`SEED_SONGS.md` notes 409s are expected and safe to skip).
- **Feed fetch failure**: if `fetchTrendingTracks` fails for one country (network error, feed schema change), that country is skipped (logged, zero candidates from it) and the run continues with the remaining countries — one country's feed being down shouldn't block the rest.
- **Whole-run failure**: only an error outside the per-candidate/per-country try/catch (e.g. the cron secret check, or a thrown error in `curateCatalog` itself) surfaces as a `500` from the route. Nothing is partially written per song — each `insertSong` call is a single atomic insert, so a mid-run crash just means fewer songs got inserted, not a corrupt row.
- **Idempotency**: `findSongByTitleArtist` dedupe means a double-fired cron run (Vercel Cron can occasionally retry) just re-skips everything already inserted earlier that day — no duplicate rows, no duplicate GPT spend for already-processed titles.
- **Cost bound**: `MAX_NEW_SONGS_PER_RUN` caps worst-case GPT-4o calls per day at a fixed number regardless of how many genuinely-new chart entries exist on a given day.

## Testing

- Manual: hit `http://localhost:3000/api/cron/curate-catalog` with `Authorization: Bearer <CRON_SECRET>` via curl before relying on the schedule, same verification pattern already used for seed scripts per `docs/SEED_SONGS.md`.
- Unit test for `fetchTrendingTracks`'s mapping logic (feed JSON shape -> `{title, artist}[]`) against a small fixture of the Apple feed's known shape.
- Unit test for `curateCatalog`'s cap/dedupe/error-isolation logic, with `findSongByTitleArtist`, `autoTagSong`, and `insertSong` mocked — verifying: cap stops further processing, a `findSongByTitleArtist` hit increments `skipped` without calling `autoTagSong`, and a thrown error from one candidate doesn't stop later candidates in the same run.
- No new tests needed for `autoTagSong`/`insertSong` themselves — unchanged, already covered.

## Addendum (post-implementation review, 2026-07-09): confidence gate before insert

The final whole-branch review (after all four implementation tasks) surfaced that `curateCatalog` had no quality gate: `autoTagSong` rarely throws (it falls back internally on iTunes/Last.fm/GPT failures rather than raising), so an unresolvable or low-confidence chart entry was inserted anyway, straight into `needs_review = true` territory, with no human review step to catch it — unlike manual seeding, where a human picks the title/artist pairs in the first place. `lib/recommend.ts` already hard-filters `final_confidence < 0.35` out of recommendations and soft-penalizes `needs_review = true`, so this wasn't a raw-garbage-in-results risk, but it meant the autonomous agent could freely add rows a human would never have chosen to seed.

Decision (confirmed with the user): `curateCatalog` now checks `tagged.needs_review` (the same boolean `autoTagSong` already computes as `final_confidence < 0.6`, `lib/autoTag.ts:406`) immediately after tagging, before calling `insertSong`. If `needs_review` is true, the candidate is **not inserted** — it's counted in `skipped`, not `failed` (it's not an error, just a "not confident enough to auto-add" outcome), and the throttle still applies (the `autoTagSong` call already happened, so the rate-limit cost was already spent). This means the autonomous agent only ever inserts songs `autoTagSong` is confident about — the same bar a human curator would implicitly apply by only seeding songs they recognize.

This does not reduce the GPT-cost-leak finding from the same review (a `needs_review` rejection still spent a real `autoTagSong` call, same as a `DuplicateSongError` rejection does) — that remains a known, accepted cost characteristic of a fully-autonomous agent with no pre-tagging dedupe-by-canonical-name check, deferred as before.

## Open Questions Deferred to Future Work

- Retiring/deprioritizing stale catalog entries that never get saved — not addressed here, catalog is append-only.
- Widening beyond Apple's chart feed (e.g. a paid trending-audio API) if chart coverage proves too narrow or too mainstream-skewed for the "Story vibe" use case.
- Making `TRENDING_COUNTRIES` / `MAX_NEW_SONGS_PER_RUN` configurable without a code change, if daily tuning turns out to be needed.
